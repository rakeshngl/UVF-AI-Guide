RAG Chat Bot

Chat with your company PDFs using a zero-cost, GPU-free AI stack: Python, ChromaDB, Docker, and Google's Gemma 4 served free via the NVIDIA API.

Welcome to another official UVF IT enterprise deployment blueprint. Feeding internal company documents (like financial reports, HR policies, or proprietary technical schematics) into public cloud AI APIs deserves a clear-eyed look at exactly what leaves your network and what doesn't — so this guide is upfront about it below.

Today, we are building a Retrieval-Augmented Generation (RAG) system where the vector database lives entirely on your own infrastructure, but the heavy AI lifting — embedding and answering — runs on NVIDIA's free, GPU-accelerated API instead of a local model. No GPU required on your end, no per-token billing, and you can "chat" with gigabytes of internal PDFs without provisioning any inference hardware.

⚠ Read this before you pick this architecture

This version trades full data locality for zero infrastructure cost. Know the difference before choosing it for sensitive documents:

  • Stays on your machine: the raw PDFs, the ChromaDB vector index, and everything at rest.
  • Leaves your machine: each text chunk (sent to NVIDIA for embedding) and each question + retrieved context (sent to NVIDIA for the Gemma 4 answer). NVIDIA's API terms govern how that data is handled on their side — read them before sending anything regulated (financial records, PII, legal contracts).
  • Want true air-gapped privacy instead? See the earlier version of this guide, which runs both embedding and generation locally via Ollama — slower and needs a GPU/CPU with headroom, but nothing leaves the network.
System Architecture Blueprint: Offline RAG Pipeline
1

Phase 1: Get a Free NVIDIA API Key

Sign up at build.nvidia.com (no credit card required), open your profile, and generate an API key. It starts with nvapi- and is valid for 6 months — put a reminder in your calendar to rotate it before then.

Create a .env file next to your project and add .env to .gitignore so the key never ends up in version control:

# .env
NVIDIA_API_KEY=nvapi-your-key-here
2

Phase 2: The Docker Infrastructure

We only need one local service now: ChromaDB, the vector database that stores your embedded PDF text. Docker Compose handles this beautifully.

Create a file named docker-compose.yml in your project folder:

services:
  chromadb:
    image: chromadb/chroma:latest
    ports:
      # Bound to localhost only — the script runs on this same machine,
      # so there's no need to expose the vector DB to the whole network.
      - "127.0.0.1:8000:8000"
    environment:
      # Without these two, Chroma silently runs in-memory and loses
      # everything on restart — even with the volume mounted below.
      - IS_PERSISTENT=TRUE
      - PERSIST_DIRECTORY=/chroma/chroma
      - ANONYMIZED_TELEMETRY=FALSE
    volumes:
      - chroma_data:/chroma/chroma
    restart: unless-stopped

volumes:
  chroma_data:

Run the stack from your terminal:

docker compose up -d

That's it for local infrastructure — no models to pull, no GPU to provision. Gemma 4 and the embedding model both run on NVIDIA's servers.

3

Phase 3: The Python Engine (LangChain)

Now we write the script that reads your PDF, chops it into readable chunks, sends them to NVIDIA for embedding, saves the vectors to the local ChromaDB server, and asks Gemma 4 questions about it.

First, install the requirements (create a virtual environment if preferred):

pip install langchain langchain-community langchain-nvidia-ai-endpoints langchain-chroma chromadb pypdf python-dotenv

Create a file named rag_agent.py and add this code:

import os
import time
from dotenv import load_dotenv
from langchain_community.document_loaders import PyPDFLoader
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_nvidia_ai_endpoints import ChatNVIDIA, NVIDIAEmbeddings
from langchain_chroma import Chroma
from langchain.chains import create_retrieval_chain
from langchain.chains.combine_documents import create_stuff_documents_chain
from langchain_core.prompts import ChatPromptTemplate

load_dotenv() # reads NVIDIA_API_KEY from your .env file

NVIDIA_API_KEY = os.environ.get("NVIDIA_API_KEY")
if not NVIDIA_API_KEY:
    raise RuntimeError(
        "Set NVIDIA_API_KEY in your .env file. Get a free key at https://build.nvidia.com"
    )

# 1. Setup connections — both models run on NVIDIA's servers, not this machine.
# A dedicated embedding model (not the chat model) for indexing,
# and Gemma 4 reserved for actually answering questions.
embeddings = NVIDIAEmbeddings(model="nvidia/nv-embedqa-e5-v5", api_key=NVIDIA_API_KEY)
llm = ChatNVIDIA(model="google/gemma-4-31b-it", api_key=NVIDIA_API_KEY, temperature=0.2)

# Connect to the LOCAL ChromaDB server container over HTTP — the vector
# index itself stays on your infrastructure even though the embedding
# calls that built it went out to NVIDIA's API.
vectorstore = Chroma(
    collection_name="company_docs",
    embedding_function=embeddings,
    host="localhost",
    port=8000,
)

# Free-tier NVIDIA endpoints cap around 40 requests/minute per model.
# Batch and pause so a big PDF doesn't get throttled mid-ingest.
BATCH_SIZE = 20
BATCH_PAUSE_SECONDS = 5

def ingest_pdf(pdf_path):
    print(f"Loading {pdf_path}...")
    loader = PyPDFLoader(pdf_path)
    docs = loader.load()

    print("Splitting document into chunks...")
    text_splitter = RecursiveCharacterTextSplitter(chunk_size=1000, chunk_overlap=200)
    splits = text_splitter.split_documents(docs)

    print(f"Embedding {len(splits)} chunks via NVIDIA API and saving to ChromaDB...")
    for i in range(0, len(splits), BATCH_SIZE):
        batch = splits[i:i + BATCH_SIZE]
        vectorstore.add_documents(batch)
        print(f"  Embedded {min(i + BATCH_SIZE, len(splits))}/{len(splits)} chunks")
        if i + BATCH_SIZE < len(splits):
            time.sleep(BATCH_PAUSE_SECONDS)

def ask_question(question):
    print(f"\nQuestion: {question}")

    system_prompt = (
        "You are an assistant for question-answering tasks. "
        "Use the following pieces of retrieved context to answer the question. "
        "If you don't know the answer, say that you don't know. "
        "\n\nContext: {context}"
    )
    prompt = ChatPromptTemplate.from_messages([
        ("system", system_prompt),
        ("human", "{input}"),
    ])

    retriever = vectorstore.as_retriever()
    question_answer_chain = create_stuff_documents_chain(llm, prompt)
    rag_chain = create_retrieval_chain(retriever, question_answer_chain)

    response = rag_chain.invoke({"input": question})
    print(f"Answer: {response['answer']}")

if __name__ == "__main__":
    pdf_file = "company_policy.pdf"

    if os.path.exists(pdf_file):
        ingest_pdf(pdf_file)
        ask_question("What is the company's remote work policy?")
        ask_question("Who is the CEO?")
    else:
        print("Please place a 'company_policy.pdf' file in this directory to test.")
4

Phase 4: Execution

Drop a PDF into the same folder as your script and name it company_policy.pdf (or change the filename in the script to match yours). Then run it:

python rag_agent.py

The script sends each chunk to NVIDIA's nv-embedqa-e5-v5 model over HTTPS, maps the text into mathematical vectors, and stores them in your local ChromaDB container. When you ask a question, it fetches only the relevant paragraphs and sends them to Gemma 4 (also on NVIDIA's servers) for a final, synthesized answer — and because the collection lives in the persistent ChromaDB container, you can re-run the script later, ask new questions, and skip re-ingesting the same PDF.

Verifying the Stack

CheckCommand / Why it matters
ChromaDB is actually persistingcurl http://localhost:8000/api/v2/heartbeat — then restart the container and confirm your collection still exists.
NVIDIA API key is validA 401/403 on first run means the key is wrong, expired, or missing from .env. Keys last 6 months — put a renewal reminder on the calendar.
Not hitting rate limitsIf ingestion errors out partway through a large PDF, increase BATCH_PAUSE_SECONDS in the script.
ChromaDB port isn't publicFrom another machine, curl http://<your-vps-ip>:8000 should fail to connect — confirms the 127.0.0.1 binding worked.
Not re-embedding on every runIngest once per PDF; re-running ingest_pdf() on the same file duplicates chunks in the collection.

What's Actually Private Here

ChromaDB runs locally inside your Docker environment, bound to localhost — your vector index and raw PDFs never leave this machine and are never reachable from the public internet. What does leave: each document chunk (sent to NVIDIA for embedding) and each question with its retrieved context (sent to NVIDIA for the Gemma 4 answer). For internal wikis, product docs, or anything not under strict data-residency requirements, this is a fast, zero-cost way to get a capable RAG assistant running. For regulated data — financial records, PII, contracts under NDA — review NVIDIA's API data-handling terms first, or use the fully local Ollama-based version of this stack instead.
